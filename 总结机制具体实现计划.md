# 总结机制具体实现计划

本文档把 `创作总结机制(不可删除).md` 细化为可执行的工程计划。若本文档、模块化实施计划和架构计划存在冲突，实现时以 `创作总结机制(不可删除).md` 为最高优先级。

## 1. 落地范围

- Module 9 已先完成总结机制的基础工具契约、内置提示词资源、创作知识模型扩展、内存实现和测试。
- Module 10 需要从旧的 executor skill 语义调整为 PromptTemplate 与 Workflow 加载系统。
- Module 11 需要接入工作流执行、控制流、反馈通信、审批、恢复和节点输出确认。
- 工作流节点编排、真实 agent 调度和审批 UI 放到后续模块继续接入。
- `创作总结机制(不可删除).md` 暂作为需求源保留在工作区，不纳入 git，除非用户明确要求。
- Outliner、Designer、Planner、Detail、Writer、Critic、Prudent、Summarizer 都是独立节点；一个节点就是一个 agent。
- 总览者、设计师、Planner 的主责是编辑对应层级的纲领文件；register 只用于长期复用知识，不自动从纲领文本抽取。

## 2. 内置资源

- 新增 `src-tauri/resources/prompt_list.json`。
  - 每一项的 key 是引用名。
  - 每一项必须包含 `prompt` 和 `describe`。
  - 默认提示词使用中文。
  - 代码不得硬编码提示词正文。
- 新增 `src-tauri/resources/display_name.json`。
  - 保存 agent、tool、operation、confirmation item 的显示名称。
  - 代码不得硬编码显示名称。
  - 代码引用 key 时用中文注释标明英文 key 的含义，便于后续本地化。
- Rust 侧新增资源加载和校验逻辑。
  - 使用 `include_str!` 加载内置 JSON。
  - 启动或测试时校验 JSON 合法、必需 key 存在、字段非空。
- PromptTemplate 的默认能力也纳入资源化管理。
  - 内置 PromptTemplate 可由 `prompt_list.json` 提供默认文本。
  - 项目级 PromptTemplate 从 `prompt_templates/` 加载。
  - 节点引用 PromptTemplate 时绑定固定版本和内容 hash。
  - 若 PromptTemplate 有新版本，只标记可更新，不自动改变旧节点行为。
- Workflow 模板从 `workflows/` 加载。
  - 模板文件名为 `workflow.json`。
  - 导入时复制展开为普通节点和连线，可编辑。
  - 源 Workflow 更新时只提示可升级，不自动同步到项目。

## 3. 创作知识模型

### 3.1 故事段

- 故事段表示作品中连续且属于同一事件的正文范围。
- 故事段正文不复制保存，只保存 `SourceSpan`。
- 故事段记录包含：
  - `segment_id`
  - 字符串小数编号，例如 `1`、`1.5`、`2`
  - 所属 `chapter_id`
  - 概括文本
  - `SourceSpan`
  - metadata
- 编号排序和中点生成使用十进制字符串算法，不使用 `f64`。
- 在两个故事段之间插入新故事段时，编号取前后编号的中点字符串。

### 3.2 事件

- 事件由一个或多个故事段组成。
- 事件状态支持：
  - `ongoing`
  - `paused`
  - `completed`
- 事件记录包含：
  - `event_id`
  - 概括文本
  - 状态
  - 关联故事段 id 列表
  - 关联章节 id 列表
  - metadata
- 事件与故事段、事件与章节都要建立双向索引。

### 3.3 人物性格与人物关系

- Planner 注册人物初始性格、人物成长、初始关系、关系变化。
- 注册项先进入 `planned` 状态。
- 章节总结阶段找到对应故事段后，注册项转为 `realized`，并建立与故事段的双向索引。
- 人物性格和人物关系变化保留完整版本路径。
- 默认查询返回最近版本。
- find 工具可查询完整变化路径。

### 3.4 伏笔

- 伏笔状态支持：
  - `planned`
  - `planted`
  - `recovered`
  - `abandoned`
- Planner 注册伏笔时创建 `planned`。
- 章节总结识别正文中已埋下伏笔后转为 `planted`。
- 后续章节回收伏笔后转为 `recovered`。
- Planner 放弃未使用伏笔时转为 `abandoned`。
- Planner 上下文默认提供全部未回收伏笔。

### 3.5 人物实体与人物出场计划

- 总览者和设计师可以决定“写什么人物”，不能只登记人物性格变化。
- 新增轻人物卡 `character_profile`，用于记录人物实体。
  - 人物名。
  - 别名列表。
  - 叙事定位。
  - 初始状态。
  - 是否新人物。
  - metadata。
- 新增人物出场计划 `character_plan`，用于记录阶段或章节中写哪些人物。
  - 人物 id 或人物名。
  - 适用范围，支持全文、阶段、章节或章节组。
  - 主写、辅助、对照等人物作用。
  - 叙事功能。
  - 出场目标。
  - 与全文主旨或阶段目标的关系。
  - metadata。
- 总览者默认规划到阶段：每阶段核心人物、对照人物、需要推进的人物功能。
- 设计师默认细化到阶段内“章节组方向”：本阶段哪些人物和关系如何推进，但不写单章细纲。
- Planner 在总览者和设计师给出的框架内生成本章大纲；如需突破框架，必须提出新的注册或纲领 patch。
- `character_trait` 只表示人物性格或状态变化，不再混用来表达“这一段该写谁”。

### 3.6 主题锚点

- 新增 `theme_anchor`，用于记录需要长期复用的主题、母题、中心思想或表达目标。
- `theme_anchor` 不替代全局总纲和阶段总纲。
- 只有需要被后续节点查询、复用、审计或保持一致性的主题信息才 register。

### 3.7 双向索引

必须支持以下双向查询：

- 故事段 <-> 章节。
- 事件 <-> 故事段。
- 事件 <-> 章节。
- 人物性格变化 <-> 故事段。
- 人物关系变化 <-> 故事段。
- 伏笔 <-> 故事段。
- 人物实体 <-> 人物计划。
- 人物计划 <-> 阶段。
- 人物计划 <-> 章节。
- 主题锚点 <-> 阶段。
- 主题锚点 <-> 章节。

索引是可重建数据，事实、摘要和注册记录才是源数据。

## 4. 工具契约

### 4.0 规划 patch 工具通用规则

- 总览者、设计师、Planner 的主责是编辑纲领文件。
- 三层纲领文件默认路径：
  - 全局总纲：`planning/global.md`。
  - 阶段总纲：`planning/stages/{stage_id}.md`。
  - 章节大纲：`planning/chapters/{chapter_id}.md`。
- 每类规划节点使用特化 patch 工具。
  - 总览者：`outliner-insert-lines`、`outliner-replace-lines`。
  - 设计师：`designer-insert-lines`、`designer-replace-lines`。
  - Planner：`planner-insert-lines`、`planner-replace-lines`。
- 每类节点只能修改自己负责的纲领文件。
  - 总览者只能修改 `planning/global.md`。
  - 设计师只能修改当前阶段对应文件。
  - Planner 只能修改当前章节对应文件。
- 节点内部可以多次调用 patch 工具，也可以被打断和继续。
- 行号 patch 使用会话累计模式：
  - 节点开始时保存 base_version、content hash 和正文/纲领快照。
  - 节点内部多次 `*-insert-lines` / `*-replace-lines` 只追加 `LinePatchOperation`。
  - 每次操作在会话模拟文本上立即应用，后续行号基于模拟文本，避免行号漂移。
  - 节点结束时一次性提交 `PatchSessionCommit`，生成最终 patch 和确认项。
- 短纲领文件允许整文件替换工具：
  - `outliner-rewrite-file`。
  - `designer-rewrite-file`。
  - `planner-rewrite-file`。
  - 正文默认不允许整文件替换，除非用户显式确认 Writer 重写模式。
- 用户确认项不按内部工具调用拆分，只按节点运行输出生成。
- 普通模式下节点输出进入待确认；Auto Mode 按节点配置自动审计、应用并 checkpoint。
- 纲领 patch 只修改纲领文件，不自动 register。
- 需要长期复用的内容必须由节点显式调用 register。

### 4.1 Outliner 工具

- Outliner 是全局总览者，仅在开局或用户明确要求重做全局规划时执行。
- Outliner 生成和维护全文主旨、讲述目标、中心思想、阶段列表和各阶段讲述内容。
- Outliner 工具：
  - `outliner-register`
  - `outliner-find`
  - `outliner-search`
  - `outliner-insert-lines`
  - `outliner-replace-lines`
- `outliner-register` 仅用于长期复用知识。
  - 人物实体。
  - 人物出场计划。
  - 人物性格变化。
  - 人物关系。
  - 伏笔。
  - 主题锚点。
- Outliner 可以创建新人物并登记轻人物卡。
- Outliner 可以为各阶段登记人物计划，但阶段总纲正文仍以 `planning/global.md` 为主数据。

### 4.2 Designer 工具

- Designer 是阶段设计师，每阶段至少执行一次，也可在阶段重规划时再次执行。
- Designer 生成和维护当前阶段总纲。
- Designer prompt 组装包含：
  - 节点提示词。
  - 全局总纲。
  - 第二次及以后生成时的既有阶段总纲。
  - 已有章节概括。
- Designer 工具：
  - `designer-register`
  - `designer-find`
  - `designer-search`
  - `designer-insert-lines`
  - `designer-replace-lines`
- `designer-register` 仅用于长期复用知识。
  - 人物实体。
  - 人物出场计划。
  - 人物性格变化。
  - 人物关系。
  - 伏笔。
  - 主题锚点。
- Designer 可以补充本阶段需要的新人物。
- Designer 默认把人物计划细化到章节组方向，不规划单章细纲。

### 4.3 Planner 工具

- `planner-register`
  - 用于注册章节大纲中需要长期复用的写作变化或计划。
  - 参数保持机制文档命名：
    - `a`: 功能。
    - `b`: 操作。
    - `c`: 内容。
  - `c` 按功能使用强类型 JSON schema。
  - 首版功能覆盖：
    - 人物实体。
    - 人物出场计划。
    - 人物性格。
    - 人物关系。
    - 伏笔。
    - 主题锚点。
  - 操作支持：
    - `list`
    - `new`
    - `update`
    - `delete`
  - `update` 仅允许更新未落地或未写入正文的计划项。
  - `delete` 仅用于删除错误操作结果，不表示角色死亡。
- `planner-find`
  - 查询本地创作知识库。
  - 默认返回编号、摘要、source、score、metadata。
  - 参数可要求附带对应正文文本。
- `planner-search`
  - 用于联网或外部 SearchProvider 考据。
  - 与本地 find 分离。
  - 搜索结果不自动入库。
- `planner-insert-lines`
  - 按 1-based 行号修改当前章节大纲文件。
- `planner-replace-lines`
  - 按 1-based 行号替换当前章节大纲文件内容。

### 4.4 Detail 工具

- Detail 是同一种节点类型的不同预设。
- 首批预设：
  - 环境。
  - 心理。
  - 设定。
- Detail 工具：
  - `detail-find`
  - `detail-search`
- Detail 可以查询细致文本和外部考据。
- Detail 输出保存到 `details/{chapter_id}.md`，仅用于留档和用户查阅。
- Writer 不通过文件路径读取 Detail 输出；细节内容由系统作为 prompt 直接传给 Writer。

### 4.5 Writer 工具

- Writer 正式写作通过行号 patch 修改正文。
- Writer 写作时允许考据现实情况，但不能注册设定。
- 正文提供给 Writer 时必须带 1-based 行号。
- Writer 工具：
  - `writer-find`
  - `writer-search`
  - `writer-insert-lines`
  - `writer-replace-lines`
- `writer-find` 查询本地创作知识，用于保持正文连续性和设定一致性。
- `writer-search` 在需要知道当前写作位置对应的现实情况时调用外部 SearchProvider。
- `writer-search` 的结果只用于当前写作判断，不自动入库，也不允许转成 register。
- 行号规则：
  - 行号从 1 开始。
  - replace 使用 `start_line` 和 `end_line` 闭区间。
  - insert 使用目标行号位置。
- 系统校验行号 patch 后，将其转换为现有 UTF-8 byte-range `DocumentPatch`。
- Writer 节点运行内多次行号操作必须进入 `PatchSession`；最终只提交一个 patch。
- 正文写回继续复用现有文档服务和 Git checkpoint。

### 4.6 Critic 工具

- Critic 是意见者节点，用来评价 Writer 生成的文字并提出意见。
- 一个 Writer 后可以连接一个或多个 Critic。
- Critic 工具：
  - `critic-find`
  - `critic-search`
- Critic 可以查询本地创作知识和外部现实资料。
- Critic 不具备 register，也不直接修改正文或纲领文件。
- Critic 输出应包含：
  - 评价对象引用。
  - 合理性问题。
  - 风格问题。
  - 设定一致性问题。
  - 节奏、人物、情节、主题表达等意见。
  - 修改建议。
- Critic 输出按每个节点运行生成确认项。

### 4.7 Prudent 工具

- Prudent 是审慎者节点，用来分析一个或多个 Critic 的意见哪些合理、哪些不合理。
- Prudent 工具：
  - `prudent-find`
  - `prudent-search`
- Prudent 可以查询本地创作知识和外部现实资料。
- Prudent 不具备 register，也不直接修改正文或纲领文件。
- Prudent 可同时连接多个 Critic。
- 多个 Critic 连接到同一个 Prudent 时，默认全部完成后运行。
- Prudent 输出应包含：
  - 合理意见列表。
  - 不合理意见列表。
  - 每条判断的理由。
  - 综合评价。
  - 返修目标。
  - 返修模式，支持 `rewrite`、`patch`、`keep`。
- Prudent 输出按每个节点运行生成确认项。
- Prudent 通过 feedback 通信边触发 Writer 返修运行。
- feedback 边必须配置 `max_communication_count`，默认 2。
- feedback 直连 Writer 只表示有限次数通信；如果返修后还要重新进入 Critic/Prudent 多轮审稿，必须显式接 Loop 节点。
- 计数耗尽后暂停并生成确认项，不继续自动通信。
- Prudent 输出被拒绝时，不触发 Writer，工作流暂停待修。

### 4.8 Summarizer 节点与职责

- 新增 `summarizer` 节点，负责章节总结流水线。
- Summarizer 不是普通工具集合；它是完整节点，内部按固定顺序执行总结步骤。
- Summarizer 负责：
  - 故事段划分并概括。
  - 事件概括。
  - 章节总结。
  - 阶段概括。
  - 检测 Planner 注册项是否在正文中落地。
- 未落地注册项进入问题队列，要求 Planner 给解释。

## 5. Find 范围

find 工具首版必须覆盖：

- 人物实体。
- 人物出场计划。
- 人物当前性格。
- 人物性格完整变化路径。
- 人物当前关系。
- 人物关系完整变化路径。
- 某事件对应的故事段。
- 某故事段对应的正文文本。
- 未回收伏笔。
- 指定伏笔的埋下和回收位置。
- 章节总结。
- 阶段总结。
- 主题锚点。

默认返回轻量结果；只有调用参数明确要求正文时才返回对应文本。

## 5.1 PromptTemplate 与模板内联

- PromptTemplate 是可复用的内联提示词模板能力，不是可执行黑盒节点。
- PromptTemplate manifest 必须包含：
  - template id。
  - name。
  - version。
  - prompt 模板。
  - describe。
  - 参数 schema。
  - metadata。
- PromptTemplate 从 `prompt_templates/` 加载。
- 内置默认提示词仍可由 `prompt_list.json` 提供。
- 节点引用 PromptTemplate 时必须绑定固定版本。
- PromptTemplate 有新版本时，只标记可更新，不自动改变旧节点行为。
- 内联调用支持参数：
  - `{{template.文风约束(风格="克制")}}`
  - GUI 后续根据参数 schema 生成表单，用户不必手写参数。
- 节点每次运行记录：
  - 原始模板 hash。
  - PromptTemplate id。
  - PromptTemplate version。
  - PromptTemplate 内容 hash。
  - 参数。
  - 输入来源映射。
  - 最终 prompt hash。
- 节点运行记录不保存展开后的完整 prompt。

## 5.2 节点提示词模板

- 每个可编辑节点配置都有“提示词”项。
- 默认提示词模板来自资源，不在代码中硬编码。
- 节点配置保存：
  - 当前提示词模板。
  - prompt key 或 PromptTemplate 引用。
  - PromptTemplate 参数。
  - 用户编辑历史备份。
- 模板持久化格式使用命名空间：
  - `{{prompt.节点提示词}}`
  - `{{input.上一章原文}}`
  - `{{template.文风约束(风格="克制")}}`
  - `{{system.当前章节号}}`
- GUI 可以显示短名，例如 `{{上一章原文}}`，但保存时必须规范化为命名空间形式。
- 数据连线的 alias 决定 `{{input.xxx}}` 的变量名。
- 缺失变量、未知 PromptTemplate、参数错误都必须返回可诊断错误，不能静默替换为空字符串。

## 5.3 Workflow 加载

- Workflow 是完整流程模板，不再叫整个流程 skill。
- Workflow manifest 文件名为 `workflow.json`。
- Workflow 从 `workflows/` 加载。
- Workflow manifest 必须声明：
  - workflow id。
  - name。
  - version。
  - 节点列表。
  - 数据边、控制流边、反馈边。
  - 依赖的 PromptTemplate 及固定版本。
  - 依赖的节点类型。
  - 需要的工具和权限。
  - 最低 Ariadne 版本。
- 导入 Workflow 时复制展开成普通节点和连线。
- 导入后的节点和连线可编辑。
- 源 Workflow 有新版本时，只提示可升级，不自动同步。
- 旧 LLM/HTTP/WASM Skill 语义迁移为 `ExecutorAdapter`，作为后续节点执行后端，不属于 PromptTemplate 或 Workflow。

## 5.4 工作流边与执行引脚

- 节点必须有执行输入和执行输出引脚，用于规范运行顺序。
- 边类型分为：
  - `data`：传 typed port 值，可带 alias，供 `{{input.xxx}}` 引用。
  - `control`：连接执行输入和执行输出，只控制运行顺序，不传业务数据。
- `feedback`：传审稿或返修通信，不作为普通文本输入。
- `feedback` 边必须携带 `FeedbackEdgeConfig { max_communication_count }`。
- 多个执行输入默认是 AND join，全部完成后运行。
- 任一触发、条件触发或循环触发必须通过 Trigger、Condition 或 Loop 节点表达。
- 审慎者与 Writer 通信必须使用 feedback 边。
- 审慎者再连审慎者或意见者时允许形成反馈链。
- feedback 直连默认最多通信 2 次；耗尽后暂停并进入确认项。
- 任何闭环必须显式接 Loop 节点，并受最大轮次、预算和超时限制。
- 没有 Loop 节点的闭环必须拒绝保存或拒绝运行。

## 6. Agent 上下文

### 6.0 Outliner 节点 / OutlinerAgent

Outliner 节点就是总览者，用于全局粒度规划。

Outliner prompt 组装包含：

- 节点提示词。
- 用户初始意图。
- 已有全局总纲。
- 已有长期知识。
- 必要时的本地查询或外部考据结果。

Outliner 输出主数据写入 `planning/global.md`。

Outliner 通过 register 记录需要长期复用的人物实体、人物计划、主题锚点、伏笔、人物变化或关系变化。

### 6.0.1 Designer 节点 / DesignerAgent

Designer 节点就是设计师，用于阶段粒度规划。

Designer prompt 组装包含：

- 节点提示词。
- 全局总纲。
- 当前阶段既有阶段总纲。
- 第二次及以后生成时的之前阶段总纲。
- 已有章节概括。
- 相关人物计划、主题锚点、伏笔和关系状态。

Designer 输出主数据写入 `planning/stages/{stage_id}.md`。

Designer 通过 register 记录需要长期复用的人物实体、人物计划、主题锚点、伏笔、人物变化或关系变化。

### 6.1 Planner 节点 / PlannerAgent

Planner 节点就是 PlannerAgent，用于生成新章节大纲。

Planner prompt 组装包含：

- 节点提示词。
- 全局总纲。
- 当前阶段总纲。
- 前文总结内容。
  - 当前阶段提供全部章节总结。
  - 非当前阶段提供阶段总结。
- 人物及人物当前性格。
- 本阶段人物计划。
- 全部未回收伏笔。
- 上一章全文。

Planner 输出主数据写入 `planning/chapters/{chapter_id}.md`。

Planner 通过 register 记录需要长期复用的人物实体、人物计划、主题锚点、伏笔、人物变化或关系变化。

### 6.2 Detail 节点 / DetailAgent

Detail 节点就是 DetailAgent，用于生成当前章节的细节材料。

- 可以有环境、心理、设定等预设。
- 可以使用 `detail-find` 和 `detail-search`。
- 输出保存到 `details/{chapter_id}.md`。
- 输出内容由系统直接塞进 Writer prompt，不要求 Writer 读文件。

### 6.3 Writer 节点 / WriterAgent

Writer 节点就是 WriterAgent，用于正式写作。

Writer prompt 组装包含：

- 节点提示词。
- 上一章原文。
- 本章大纲。
- 本章细节。
- 审慎者返修上下文。
- 带行号的当前章节正文或待写入位置上下文。

Writer 可以使用 `writer-find` 和 `writer-search` 做本地查询与现实考据。
Writer 默认不接收可回收伏笔类大纲信息。
Writer 不具备 register 工具，不能注册人物性格、人物关系或伏笔。

Writer 默认提示词模板形态：

```text
{{prompt.节点提示词}}
以下是上一章的原文，你需要根据其的风格行文方式等等依照本章大纲继续写：{{input.上一章原文}}
本章大纲：{{input.本章大纲}}
```

### 6.4 Critic 节点 / CriticAgent

Critic 节点就是意见者，用于评价 Writer 生成的文字。

Critic prompt 组装包含：

- 节点提示词。
- 待评价文本。
- 本章大纲。
- 阶段总纲。
- 相关人物计划。
- 相关章节总结或前文摘要。

Critic 输出进入确认项。

### 6.5 Prudent 节点 / PrudentAgent

Prudent 节点就是审慎者，用于分析一个或多个意见者输出。

Prudent prompt 组装包含：

- 节点提示词。
- 一个或多个 Critic 输出。
- 待评价文本。
- 本章大纲。
- 阶段总纲。
- 相关人物计划。
- 相关章节总结或前文摘要。

Prudent 输出进入确认项。

Prudent 通过 feedback 边给 Writer 发送 `revision_context`，触发 Writer 返修运行。

## 7. 章节总结流水线

触发时机：

- 正式写作通过审核后触发。

执行顺序：

1. 故事段划分并概括。
   - 为故事段编号。
   - 默认使用上一个编号的下一个整数。
   - 手动插入故事段时使用字符串中点编号。
   - 建立故事段与章节的双向索引。
2. 事件概括。
   - 将多个故事段归入同一事件。
   - 或将故事段加入已有事件。
   - 有变动的事件重新生成概括和状态。
   - 建立事件与故事段、事件与章节的双向索引。
3. 章节总结。
   - 总结当前章节写了什么。
   - 写出各事件进展。
   - 输入包含章节内容、事件变动对比、人物性格和关系变动对比。
4. 阶段概括。
   - 判断本章是否仍属于当前阶段。
   - 若不是，LLM 提案创建新阶段。
   - 普通模式需要确认或按配置跳过。
   - Auto Mode 默认自动审计。
   - 无论是否新阶段，都更新包含本章的阶段总结。

## 8. 确认项和 Auto Mode

确认项包括：

- Outliner 节点输出。
- Designer 节点输出。
- Planner 节点输出。
- planner-register 的所有子功能。
- 规划 patch 节点输出。
- Critic 节点输出。
- Prudent 节点输出。
- 故事段划分并概括。
- 事件概括。
- 章节总结。
- 阶段概括。
- Writer 自动修正文档 patch。

普通模式：

- 每类确认项都可以独立配置为人工确认或跳过。
- 默认需要人工确认。

Auto Mode：

- 默认对确认项执行自动审计。
- 每类确认项可配置为自动跳过或自动审计。
- 自动审计 prompt 从 `prompt_list.json` 读取。
- 成本控制和硬权限仍然可以阻断运行。

确认或审计失败：

- 暂停当前流水线。
- 记录待处理项。
- 等待修正或人工处理。

确认粒度：

- 规划 patch 不按节点内部工具调用拆确认项。
- Outliner、Designer、Planner 的规划 patch 按每个节点运行输出生成确认项。
- Critic 和 Prudent 按每个节点输出生成确认项。
- Prudent 输出被拒绝时，不触发 Writer 返修。
- Writer 返修输出仍按 Writer 输出或 patch 规则进入确认项。

## 9. Planner 与 Summarizer 通信

- 未落地注册项会生成问题队列项。
- 问题队列项包含：
  - 注册项 id。
  - 预期变化。
  - Summarizer 未匹配到的原因。
  - 相关章节、故事段、正文引用。
- Planner 需要针对问题给出解释。
- Planner 可给出的处理结果：
  - 改变想法：建议将注册项标记为 `abandoned`。
  - 写在别处：指出具体故事段或章节位置。
  - 确实忘了写：生成补写建议和目标。
  - Summarizer 误判：说明已有落地证据。
- 所有处理结果都进入确认项。
- 若确实需要补写：
  - 普通模式必须先由用户确认补写建议。
  - Auto Mode 可生成补写建议和候选 patch，但只进入待审队列，不自动写回已审核正文。
  - Writer 只有在确认通过后才根据建议生成补写 patch。
  - patch 通过正常审批或自动审计后才写回正文。
  - 写回后从受影响步骤重新跑章节总结流水线。
- 禁止未经确认自动修改已审核通过正文。

- 审查报告中的 Hybrid 候选数量复核已落地：
  - 混合检索增加 `MAX_HYBRID_SEARCH_LIMIT` 和 `MAX_HYBRID_CANDIDATE_LIMIT`。
  - 超出最大返回数量的请求直接校验失败，避免无界 `limit` 进入底层索引和 reranker。

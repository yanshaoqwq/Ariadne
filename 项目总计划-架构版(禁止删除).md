## 一、项目背景与目标

### 1.1 核心定位

Literature Agent 是面向长篇小说创作的可视化 AI Agent Harness 桌面系统。核心目标是通过多智能体协作、长文本上下文工程、审批控制和 Git 版本管理，增强 LLM 在长篇创作中的稳定性、可追溯性和可控性。

### 1.2 设计原则

- **完整 MVP**: MVP 必须覆盖写作、检索、工作流、Skill、搜索、审批、导出、Git、成本控制等核心闭环，不把关键能力推迟到后续版本。
- **至少百万字可用**: 真实百万字项目必须可用；架构按任意长度扩展设计。
- **最佳性能优先**: 后台任务、多级索引、缓存、混合检索、批处理和异步执行作为默认设计，不使用会阻碍长文本规模的简化路径。
- **模块契约先行**: 先定义 Module 0 核心契约，后续模块只新增能力并接入契约，避免反复大改已有模块。
- **安全硬限制**: API Key 使用系统密钥链；WASM 脚本、路径访问、密钥读取和预算上限始终受硬限制保护。
- **成本可控**: 普通模式逐次确认高成本操作；Auto Mode 使用预算预授权，超限暂停。
- **可恢复执行**: 工作流支持 Pause、Stop、断点恢复、节点级 checkpoint 和 Git 存档点。

### 1.3 目标规模

- **MVP 最低目标**: 百万字真实小说项目人工验收可用。
- **扩展目标**: 任意长度项目通过分层摘要、冷热索引、增量重建和引用式数据流扩展。
- **索引策略**:
  - chunk/场景级: 精细内容检索。
  - 章节级: 章节摘要、关键事件、角色出场。
  - 卷级/全书级: 长期记忆、全局设定、跨章节一致性。
  - 冷热数据: 热数据保留高频索引和缓存，冷数据按需召回和重排。

## 三、核心业务逻辑

### 3.0 节点类型定义

**MVP 内置节点类型:**

1. **LLMNode**
   - 输入: prompt template、上下文引用、参数、工具配置。
   - 输出: 文本、patch、artifact 引用、工具调用日志、成本信息。
   - 功能: 调用 OpenAI / Anthropic / Gemini 协议模型，支持流式输出和 tool-use。

2. **DocumentNode**
   - 输入: 文档路径、范围、patch、artifact 引用。
   - 输出: 文档引用、选区引用、文档元数据、diff。
   - 功能: Markdown、txt、JSON 的读取、写入、patch 预览和增量索引触发。

3. **SkillNode**
   - 输入/输出: 根据 Skill schema 暴露 typed ports。
   - 功能: 执行 LLM、HTTP、WASM Script Executor。

4. **SearchNode**
   - 输入: 查询、过滤条件、来源范围。
   - 输出: 搜索结果、引用、评分、来源元数据。
   - 功能: 本地知识库搜索 + 可配置 Web SearchProvider。

5. **Condition / Eval Node**
   - 输入: 上游输出、评分标准、审批结果。
   - 输出: boolean、score、reason、分支选择。
   - 功能: 判断是否进入循环、是否需要人工或自动审批。

6. **Loop / Trigger Node**
   - 输入: 条件、最大轮次、预算上限、停止策略。
   - 输出: 当前轮次、循环状态、终止原因。
   - 功能: 支持受限循环，不支持无界循环。

7. **Approval Node（审批模块）**
   - 输入: 待审批内容、审批提示词、上下文引用。
   - 输出: approved、rejected、review_reason、reviewer_model_output。
   - 功能: 普通模式人工确认；Auto Mode 下按节点配置自动审批。

8. **Export Node**
   - 输入: 文档集合、目录结构、导出配置。
   - 输出: EPUB/PDF artifact 引用。
   - 功能: MVP 支持基础 EPUB/PDF 导出。

### 3.1 Typed Ports 与大数据传递

所有节点使用显式 typed input/output ports。大文本和大结果禁止默认内联传递，必须优先使用引用:

```typescript
type PortValue =
  | { kind: "inline"; value: unknown }
  | { kind: "document_ref"; documentId: string; range?: TextRange }
  | { kind: "chunk_ref"; chunkId: string }
  | { kind: "artifact_ref"; artifactId: string };
```

这样可以避免百万字项目中节点状态、运行日志和 checkpoint 体积失控。

### 3.2 配置管理（Module 1）

配置采用分文件 YAML，并全部带 schema version:

```
.config/
├── app.yaml
├── providers.yaml
├── permissions.yaml
├── rag.yaml
├── workflow.yaml
├── git.yaml
└── auto_mode.yaml
```

- 密钥存储在全局系统 keychain，项目配置只保存 key id。
- 数据库和配置文件都必须有版本化迁移机制。
- 支持 OpenAI、Anthropic、Gemini 三类协议格式；OpenAI-compatible `base_url` 作为 OpenAI 格式子类支持本地服务。
- 迁移必须幂等、可记录、可重试；破坏性升级前创建备份，失败后保留旧数据并进入可诊断状态。
- 大型项目迁移必须分批执行并报告进度，避免启动或 UI 长时间阻塞。

### 3.3 成本追踪（Module 2）

- 所有 LLM、Embedding、Reranker、Search API、冲突审批调用都进入成本账本。
- 普通模式在高成本操作前确认。
- Auto Mode 使用单次、每日、月度预算预授权，超限自动 Pause。
- 成本预估允许显示区间或上限，不承诺 tool-use 多轮调用的绝对准确值。

### 3.4 检索与索引（Module 4 / 8 / 9）

**默认后端:**
- VectorStore: 应用托管本地 Qdrant sidecar。
- FullTextStore: Tantivy。
- StructuredStore: SQLite。
- Reranker: 可插拔，默认启用，可配置本地或远程 provider。

**Qdrant sidecar 生命周期:**
- 应用负责二进制发现、版本检查、数据目录隔离、端口选择、启动、健康检查、日志采集、崩溃检测、重启和优雅关闭。
- 端口冲突时自动选择可用端口或提示用户配置外部 Qdrant。
- Qdrant 不可用时 VectorStore 报告 degraded/unavailable，不允许 RAG 静默返回空结果。
- Qdrant/Tantivy 索引属于可重建数据；损坏时应从源文档、metadata 和 Git 状态重建。

**索引架构:**
- 分层索引: chunk、场景、章节、卷、全书。
- 增量索引: 只重建变更文档、变更章节及受影响摘要。
- 冷热策略: 热路径保留高频索引和缓存，冷数据按需加载。
- 后台任务: 保存、Git 回档、导入、重建索引都进入后台 worker 队列。

**RAG 工具设计:**
- 少工具强参数，不拆成大量专用工具。
- 同一个工具通过 `operation` 参数执行语义搜索、结构化查询、邻近上下文、摘要查询等子操作。
- 检索流程: 向量召回 + 全文召回 + 结构化过滤 + reranker + 引用返回。

### 3.5 知识库与长期记忆（Module 8）

- 结构化数据包括角色、别名、关系、世界观、时间线、章节摘要、冲突项。
- 默认流程: AI 抽取候选数据，普通模式人工确认后入库。
- Auto Mode: 自动通过普通确认环节，但冲突不自动覆盖。
- 冲突处理:
  1. 写作 LLM 给出“为什么这样写合理”的理由。
  2. 独立判断提示词的 LLM 审批理由和冲突。
  3. 审批结果、理由、来源引用进入待审队列和运行日志。
- 所有摘要和结构化事实必须保留来源文档、范围和版本。

### 3.6 Git 服务（Module 5）

- Git 健康检查以检测、备份、降级和重新初始化为主，不承诺 `git fsck` 自动修复所有损坏。
- Git 操作全局互斥。
- 用户可创建命名存档点。
- 节点可创建 checkpoint commit。
- Pause 和 Stop 都不回滚已经产生的 Git 历史。
- 回档时创建新分支保护当前工作，并触发索引重建。

**默认跟踪策略:**
- 文档、工作流、Skill、非敏感配置是否入库由配置决定。
- 向量库、全文索引、缓存、成本库、运行状态库默认忽略。
- 运行 checkpoint 默认保存在本地状态库，可配置提交可读摘要到 Git。

### 3.7 Skill 系统（Module 10）

**MVP Executor:**
- LLM Executor
- HTTP Executor
- WASM Script Executor

不支持不受控本机 Python/JavaScript 脚本。WASM 脚本必须受路径、网络、时间、内存和能力白名单限制。

Skill 支持全局库和项目库；项目 Skill 优先。Skill 暴露 input_schema、output_schema 和 typed ports，可在工作流中直接连接。

### 3.8 工作流引擎（Module 11）

工作流不是纯 DAG。MVP 支持受限循环:

- 每个循环必须有最大轮次。
- 每个循环必须受预算和超时限制。
- 每个循环必须有终止条件或审批节点。
- 支持 Pause: 保存运行态并可继续。
- 支持 Stop: 终止当前运行，保留已完成节点、Git 历史、artifact 和 checkpoint。
- 支持断点恢复: 失败或暂停后从最近安全点继续。

节点输出、工具调用、审批意见、成本、错误、恢复点都进入运行日志。

运行状态持久化必须保存 PortValue 引用、artifact 引用、checkpoint、预算消耗和待审批状态。Resume 时对已完成节点保持幂等，不能重复计费或重复写回。恢复时必须校验引用目标是否存在；引用丢失、runtime.db 损坏、provider 中断或 sidecar 崩溃时进入可诊断失败或降级状态，而不是静默跳过。

### 3.9 Auto Mode 与审批

- 全局 Auto Mode 只表示跳过普通人工确认流，不修改权限开关配置值。
- 系统执行时先判断 Auto Mode；若开启则普通确认直接放行。
- 预算、路径沙箱、WASM 沙箱、密钥保护等硬限制仍强制执行。
- 审批模块由节点级配置控制是否自动审批，以及使用哪个审批提示词。
- 审批 LLM 的输出必须作为显式输出引脚，可供下游节点使用。

## 四、前端架构

### 4.1 技术栈

- Tauri 2 当前稳定主线

## 五、数据模型

### 5.1 项目结构

```
my-novel/
├── .config/
│   ├── app.yaml
│   ├── providers.yaml
│   ├── permissions.yaml
│   ├── rag.yaml
│   ├── workflow.yaml
│   ├── git.yaml
│   └── auto_mode.yaml
├── .git/
├── .cache/                 # 默认不入 Git
├── .runtime/               # 运行状态与 checkpoint，默认不入 Git
├── .indexes/               # Qdrant/Tantivy 数据，默认不入 Git
├── .knowledge/             # SQLite metadata，默认不入 Git
├── documents/
├── workflows/
├── skills/
└── exports/
```

### 5.2 本地数据库

- `metadata.db`: 角色、世界观、时间线、摘要、冲突队列。
- `costs.db`: API 调用、预算、预授权。
- `runtime.db`: 工作流运行、节点状态、Pause/Stop、恢复点。

三类数据库分开管理，均带 schema version 和迁移记录。

### 5.3 迁移与恢复策略

- 配置文件、SQLite 数据库、索引元数据和运行状态都必须有 schema version。
- 迁移记录包含组件名、from/to version、执行时间、状态和错误信息。
- 破坏性迁移前创建备份或快照；失败后保留旧数据，允许用户重试或回滚到备份。
- 索引和缓存不作为唯一事实源，损坏后从源文档、metadata、Git 状态和运行日志重建。
- Git 损坏以检测、备份、降级和重新初始化为主，不承诺自动修复所有损坏。

### 5.4 文档格式

MVP 支持:
- Markdown
- txt
- JSON

docx/epub 作为导入导出方向，不作为 MVP 编辑格式。

### 5.5 Artifact 与 Patch

LLM/Skill 默认输出 patch/diff 或 artifact 引用。普通模式预览确认后写回；Auto Mode 按节点配置自动应用并 checkpoint。

## 六、安全、隐私与审计

- API Key 只存在系统 keychain；项目文件只保存 key id。
- 文件访问必须经过路径沙箱，并防止父目录逃逸和符号链接逃逸。
- 网络访问、Web 搜索、HTTP Skill、WASM 能力必须由权限配置显式允许。
- 运行日志和导出内容默认不得包含密钥、完整认证头、敏感环境变量或系统隐私信息。
- 审计日志记录配置变更、权限变更、自动审批、预算放行、Git 回档、外部网络调用和恢复操作。
- Auto Mode 不能绕过预算、路径沙箱、WASM 沙箱、密钥保护和网络权限。

## 七、性能监控与诊断

- 后台索引、检索、reranker、工作流恢复、导出都输出耗时、队列长度、错误和资源使用指标。
- 百万字固定测试数据集作为性能回归基准，覆盖导入、索引、检索、写回、恢复和导出。
- 前端诊断面板展示索引进度、sidecar 健康、后台任务、成本、慢操作、错误和恢复建议。
- Provider、Qdrant sidecar、Tantivy、SQLite、Git、keychain 都暴露健康状态或诊断信息。

---

## 八、分发与依赖管理

- Qdrant sidecar 需要跨平台打包、版本校验、日志目录和数据目录策略。
- Tantivy 索引、SQLite 数据库、keychain 后端、导出工具链需要明确的兼容性和升级策略。
- 本地依赖不可用时，应用应提示原因，并支持配置外部服务或进入功能降级状态。
- Provider 真实 API 测试必须通过显式环境变量启用，默认测试使用 mock 或本地兼容服务，避免产生不可控费用。

## 九、实施路线图

### Phase 0: 技术验证

目标是验证高风险基础能力:

1. Qdrant sidecar 生命周期、分发、百万字向量检索。
2. Tantivy 中文全文检索与混合召回。
3. 可插拔 reranker 延迟与成本。
4. 受限循环、Pause、Stop、恢复原型。
5. WASM Skill 沙箱权限模型。
6. keychain 跨平台验证。
7. 迁移失败回滚、索引重建、runtime 恢复和 Git 降级流程。
8. 百万字数据集性能基准和前端诊断链路。

### Phase 1: 完整 MVP

MVP 包含 Module 0-12 的完整闭环:

## 十、关键决策与权衡

### 7.1 为什么保留受限循环？

小说创作常见“生成 - 评估 - 重写 - 审批”的闭环。完全移除循环会削弱 Agent Harness 的核心价值。MVP 支持受限循环，但必须有最大轮次、预算、超时、终止条件和可恢复状态。

### 7.2 为什么目标改为至少百万字？

长篇小说创作的核心痛点正是长期上下文和跨章节一致性。10 万字只能覆盖早期项目，不能证明系统能力。MVP 必须以真实百万字项目人工验收为最终标准。

### 7.3 为什么使用少工具强参数？

大量 RAG 工具会增加 token 成本和模型选择错误概率。使用少量工具配合 `operation` 和强 schema，可以保留能力，同时降低工具定义负担。

### 7.4 为什么 Qdrant 使用 sidecar？

当前计划不假设 Rust 客户端具备可靠的嵌入式库模式。使用应用托管的本地 sidecar 可以保持桌面体验，同时通过 `VectorStore` 抽象保留替换空间。

### 7.5 为什么 Skill 脚本使用 WASM？

脚本能力是完整 Skill 系统的一部分，但本机脚本安全风险过高。WASM 沙箱提供更明确的能力边界。

## 十一、风险与缓解

- **百万字性能风险**: 使用分层索引、后台 worker、冷热缓存、混合召回和自动性能回归。
- **成本风险**: 预算预授权、硬限制、成本账本、Auto Mode 超限 Pause。
- **知识污染风险**: AI 抽取先入候选，冲突进入两步 LLM 审批和待审队列。
- **Git 噪声风险**: 文档/工作流/Skill 入库可配置，缓存和数据库默认忽略。
- **脚本安全风险**: WASM 沙箱、能力白名单、时间和内存限制。
- **工作流死循环风险**: 循环必须有最大轮次、预算、超时和终止条件。
- **迁移失败风险**: schema version、迁移记录、升级前备份、失败可重试或回滚。
- **索引损坏风险**: Qdrant/Tantivy 索引可检测、可丢弃、可从源文档和 metadata 重建。
- **sidecar 分发风险**: 版本检查、端口冲突处理、健康检查、日志采集和外部服务配置兜底。
- **隐私泄露风险**: 密钥只进 keychain，日志脱敏，网络和文件能力默认最小授权。

## 十二、集成测试场景

1. 百万字真实项目导入、索引、检索、写作、回档人工验收。
2. 混合 RAG: 向量 + 全文 + 结构化 + reranker。
3. 受限循环: 生成、评估、不通过重写、达到条件退出。
4. Pause/Resume: 暂停后从运行态继续。
5. Stop: 终止后保留 Git、artifact、已完成节点和 checkpoint。
6. Skill: LLM、HTTP、WASM 三类 executor。
7. Auto Mode: 跳过普通确认，但预算和沙箱硬限制生效。
8. 知识冲突: 写作理由 + 独立审批 + 待审队列。
9. Git: 存档点、节点 checkpoint、回档后索引重建。
10. Export: Markdown 项目导出 EPUB/PDF。
11. 迁移: 旧 schema 项目升级、失败回滚、重复迁移幂等。
12. 恢复: 索引损坏、runtime.db 损坏、provider 中断、sidecar 崩溃后的恢复或降级。
13. 安全: 路径穿越、符号链接逃逸、HTTP/WASM 越权、日志敏感信息脱敏。
14. 性能: 固定百万字数据集导入、索引、检索、写回、恢复和导出回归。

## 十三、验收标准

### 13.1 功能完整性

- [ ] 可创建和运行包含 LLM、Document、Skill、Search、Approval、Loop、Export 的工作流。
- [ ] 支持受限循环、Pause、Stop、断点恢复。
- [ ] 支持 Markdown/txt/JSON 编辑。
- [ ] 支持 patch 预览与写回。
- [ ] 支持 Git 存档点和节点 checkpoint。
- [ ] 支持基础 EPUB/PDF 导出。

### 13.2 长文本能力

- [ ] 真实百万字项目人工验收可用。
- [ ] 索引、检索、写作、回档、恢复流程不阻塞主 UI。
- [ ] 自动性能回归覆盖索引、检索、reranker、恢复。

### 13.3 安全与成本

- [ ] 密钥不进入项目文件。
- [ ] Auto Mode 不绕过预算、路径沙箱、WASM 沙箱和密钥保护。
- [ ] 路径穿越、符号链接逃逸、HTTP/WASM 越权均被拒绝。
- [ ] 日志和导出内容默认不包含密钥、完整认证头或敏感环境变量。
- [ ] 所有 API 调用进入成本账本。
- [ ] 超预算时工作流 Pause。

### 13.4 恢复与迁移

- [ ] 配置、metadata.db、costs.db、runtime.db 都有版本化迁移和迁移记录。
- [ ] 破坏性升级前创建备份，失败后可重试或回滚。
- [ ] Qdrant/Tantivy 索引损坏可检测、可重建。
- [ ] Resume 对已完成节点幂等，不重复计费或重复写回。

### 13.5 用户体验

- [ ] 编辑器和节点画布并重。
- [ ] 运行日志能查看节点状态、工具调用、成本、错误和恢复点。
- [ ] 知识冲突和审批意见可审计。
- [ ] 前端能查看 sidecar 健康、索引进度、后台任务、性能指标和恢复入口。
- [ ] 首次配置、API Key、权限、Git、导出有基础用户引导。

## 十五、总结

v4.0 的核心变化是: MVP 不再是瘦身版本，而是完整长篇小说 Agent Harness 的第一版可用系统。架构必须优先支持百万字真实项目、受限循环、完整 Skill、审批与 Auto Mode、混合 RAG、Git 可追溯和极致性能。

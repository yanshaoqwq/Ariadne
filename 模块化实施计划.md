## 横向补充策略

以下策略来自项目检验报告中合理的缺失项，作为所有后续模块的共同约束。

### 数据迁移与升级

- 配置、SQLite 数据库、索引元数据、运行状态库都必须有 schema version。
- 迁移必须幂等；破坏性迁移前必须创建备份或可回退快照。
- 迁移记录必须包含组件名、from/to version、执行时间、状态和错误信息。
- 升级失败时不得继续启动到不一致状态；应用应保留旧数据、报告错误，并允许用户重试或回滚到备份。
- 大型项目迁移必须分批执行，并向前端报告进度，避免阻塞主 UI。

### 错误恢复与灾难恢复

- 索引、缓存、运行状态、Git、provider、sidecar 服务都必须暴露健康检查。
- 可重建数据优先从源文档、Git 历史和结构化 metadata 重建，不把索引视为唯一数据源。
- 工作流恢复必须基于持久化的节点状态、输出引用、artifact、成本记录和 checkpoint。
- 恢复失败必须降级为可诊断状态，禁止静默丢失已完成结果。

### 安全、隐私与审计

- API Key 只存在系统 keychain；项目文件只保存 key id。
- 文件访问必须经过路径沙箱，网络访问、Web 搜索、HTTP Skill、WASM 能力必须由权限配置显式允许。
- 运行日志和导出内容不得默认写入密钥、完整请求头或敏感系统信息。
- 关键操作需留下审计信息: 配置变更、权限变更、自动审批、预算放行、Git 回档、外部网络调用。

### 性能监控与诊断

- 后台索引、检索、reranker、工作流恢复、导出都必须输出耗时、队列长度、错误和资源使用指标。
- 百万字测试数据集作为固定回归基准；性能回归需覆盖导入、索引、检索、写回、恢复。
- 前端需要提供运行诊断视图，展示索引进度、sidecar 健康、成本、错误和慢操作。

### 测试策略

- 单元测试覆盖模块内部边界、错误路径和安全约束。
- 集成测试覆盖跨模块契约、恢复路径、权限拒绝和成本入账。
- Provider 真实 API 测试必须通过显式环境变量启用，默认使用 mock 或本地兼容服务。
- 性能测试使用可复现数据集和固定阈值，避免把真实 API 成本放进默认测试。

### 分发与依赖

- Qdrant sidecar、Tantivy 索引、keychain 后端、导出工具链都需要跨平台打包和版本检查策略。
- 本地服务端口、数据目录、日志目录和进程生命周期必须可诊断。
- 如果默认本地依赖不可用，应用应明确提示并支持配置外部服务或降级路径。

## Module 0: 核心契约与公共基础

**目标**: 先固定跨模块契约，使后续模块只新增实现并接入公共接口。

### 0.1 模块职责

- 定义 Node、Edge、Workflow、Port、Artifact、Provider、Skill 的公共数据结构。
- 定义 typed input/output ports，支持多输入多输出。
- 定义大文本引用传递，禁止百万字内容在节点间默认内联复制。
- 定义模块注册表: NodeRegistry、SkillRegistry、ProviderRegistry。
- 定义统一错误模型、事件模型、运行状态、资源池和取消信号。
- 定义 Auto Mode、审批、权限硬限制的公共语义。

### 0.2 文件结构

```
src-tauri/src/core/
├── mod.rs
├── ports.rs
├── workflow.rs
├── artifacts.rs
├── registry.rs
├── events.rs
├── errors.rs
├── resources.rs
└── permissions.rs
```

### 0.3 核心契约

```rust
pub enum PortValue {
    Inline(serde_json::Value),
    DocumentRef { document_id: String, range: Option<TextRange> },
    ChunkRef { chunk_id: String },
    ArtifactRef { artifact_id: String },
}

pub struct PortDefinition {
    pub name: String,
    pub type_name: String,
    pub required: bool,
}

pub struct NodeDefinition {
    pub type_name: String,
    pub input_ports: Vec<PortDefinition>,
    pub output_ports: Vec<PortDefinition>,
    pub supports_checkpoint: bool,
    pub supports_auto_approval: bool,
}

pub enum RunControl {
    Continue,
    Pause,
    Stop,
}

pub struct LoopPolicy {
    pub max_iterations: u32,
    pub timeout_ms: u64,
    pub budget_limit_usd: Option<f64>,
    pub stop_condition: serde_json::Value,
}
```

### 0.4 Auto Mode 语义

- Auto Mode 不修改权限配置项。
- 执行时先判断 Auto Mode；开启时跳过普通人工确认。
- 预算、路径沙箱、WASM 沙箱、密钥保护始终是硬限制。
- 审批行为由节点级配置决定；全局只提供可用审批提示词配置。

### 0.5 验收标准

- [ ] 所有模块只依赖 Module 0 类型和接口通信。
- [ ] typed ports 支持审批意见、正文、patch、artifact、成本等独立输出。
- [ ] 大文本通过引用传递。
- [ ] Pause 与 Stop 语义在核心状态模型中明确区分。

## Module 1: 配置管理、密钥存储与迁移

**目标**: 管理项目配置、全局密钥、配置 schema 和迁移。

### 1.1 模块职责

- 读取和写入分文件 YAML 配置。
- 使用系统 keychain 存储全局密钥，项目仅引用 key id。
- 配置文件带 schema version，启动时自动迁移。
- 支持 provider、permissions、rag、workflow、git、auto mode 等配置。
- 提供配置和数据库迁移的统一策略: 幂等、可记录、可重试、失败不破坏旧数据。

### 1.2 配置布局

```
.config/
├── app.yaml
├── providers.yaml
├── permissions.yaml
├── rag.yaml
├── workflow.yaml
├── git.yaml
└── auto_mode.yaml
```

### 1.3 关键配置

- `providers.yaml`: OpenAI、Anthropic、Gemini、OpenAI-compatible base_url、Embedding、Reranker、SearchProvider。
- `permissions.yaml`: 网络、Web 搜索、HTTP Skill、WASM 能力、文件路径白名单。
- `auto_mode.yaml`: Auto Mode 默认值、预算预授权、可用审批提示词。
- `git.yaml`: 文档、工作流、Skill、配置是否入库；缓存和数据库默认忽略。

### 1.4 验收标准

- [ ] 密钥不落项目文件。
- [ ] 配置和数据库迁移均可重复执行。
- [ ] 破坏性配置升级前有备份，失败后可以回滚或重试。
- [ ] 大型项目升级能报告进度，且不会阻塞主 UI。
- [ ] Auto Mode 配置不会改写权限开关。
- [ ] OpenAI-compatible 本地服务可通过 base_url 配置。

## Module 2: 成本追踪与预算控制

**目标**: 所有模型和外部 API 调用可追踪、可预估、可限制。

### 2.1 模块职责

- 记录 LLM、Embedding、Reranker、Search API、HTTP Skill、冲突审批调用。
- 支持单次、每日、月度预算。
- 普通模式执行前确认高成本操作。
- Auto Mode 在预算预授权内自动执行，超限触发 Pause。

### 2.2 数据存储

- `costs.db`: API 调用、模型、tokens、费用、workflow_id、node_id、tool_call_id。
- 预算配置来自 `.config/auto_mode.yaml` 和 `.config/providers.yaml`。

### 2.3 验收标准

- [ ] 所有 provider 调用都经过成本记录。
- [ ] tool-use 多轮调用按实际发生费用入账。
- [ ] 未知 tool-use 轮次时成本预估显示区间或上限，不展示为精确值。
- [ ] 超预算时工作流 Pause，而不是静默失败。

## Module 4: 向量、全文与重排检索

**目标**: 提供高性能混合检索基础。

### 4.1 模块职责

- VectorStore 抽象，默认应用托管本地 Qdrant sidecar。
- FullTextStore 抽象，默认 Tantivy。
- Reranker 抽象，默认启用，可选本地或远程 provider。
- 支持后台批量写入、增量删除、冷热层索引。

### 4.2 实施要求

- 不假设 Rust SDK 直接嵌入 Qdrant。
- sidecar 生命周期由应用管理: 启动、健康检查、停止、端口冲突处理。
- sidecar 必须支持二进制发现、版本检查、数据目录隔离、日志采集、崩溃检测、指数退避重启和优雅关闭。
- 端口冲突必须自动选择可用端口或明确提示用户配置外部 Qdrant。
- Qdrant 不可用时，VectorStore 必须报告 degraded/unavailable 状态，不能让 RAG 静默返回空结果。
- 检索返回必须包含来源引用和 score。

### 4.3 验收标准

- [ ] 百万字项目混合检索可用。
- [ ] 主 UI 不因索引或检索阻塞。
- [ ] VectorStore 可替换，不影响 RAG 和知识库模块。
- [ ] Qdrant sidecar 启动、端口冲突、崩溃重启、健康检查和关闭路径均有测试。
- [ ] Tantivy/Qdrant 索引损坏时可检测，并能从源文档和 metadata 重建。

## Module 5: Git 服务与版本控制

**目标**: 提供可追溯、可恢复、可配置的 Git 集成。

### 5.1 模块职责

- 初始化和健康检查 Git 仓库。
- 创建用户命名存档点。
- 创建节点级 checkpoint commit。
- 支持 diff、分支图、回档到新分支。
- 回档后触发索引重建。

### 5.2 跟踪策略

- 文档、工作流、Skill、非敏感配置是否入库由配置决定。
- 缓存、向量库、全文索引、成本库、运行状态库默认忽略。
- Stop 不回滚已有 Git 历史。

### 5.3 验收标准

- [ ] Git 操作全局互斥。
- [ ] 回档不会破坏当前工作，必须创建新分支。
- [ ] Git 损坏时支持检测、备份、降级和重新初始化，不承诺自动修复所有损坏。
- [ ] 回档后索引和运行状态能标记为待重建或待重新绑定。

## Module 6: 文档服务、Artifact 与 Patch 写回

**目标**: 高性能管理 Markdown、txt、JSON 和生成结果写回。

### 6.1 模块职责

- 支持 Markdown、txt、JSON。

### 6.2 验收标准

- [ ] 大文本不在运行状态中重复复制。
- [ ] patch 写回和 Git checkpoint 可联动。
- [ ] 路径穿越、符号链接逃逸和越权写入均被拒绝。

## Module 8: 知识库、分层索引与长期记忆

**目标**: 支撑至少百万字小说的长期上下文。

### 8.1 模块职责

- 文档 chunk、场景、章节、卷、全书分层摘要。
- 角色、别名、关系、世界观、时间线、章节摘要结构化数据。
- AI 抽取候选数据，普通模式人工确认。
- Auto Mode 自动通过普通确认，但冲突不覆盖。
- 冲突队列和两步 LLM 审批。
- 结构化 metadata 和摘要版本必须支持迁移、备份和校验；可从源文档重建的索引不得作为唯一事实源。

### 8.2 冲突审批

1. 写作 LLM 输出“为什么这样写合理”的理由。
2. 独立判断提示词的 LLM 审批。
3. 审批结果、理由、来源进入待审队列。

### 8.3 验收标准

- [x] 每条摘要和事实都保留来源和版本。
- [x] 冲突不会静默覆盖旧设定。
- [x] Auto Mode 不阻塞，但待审信息完整。
- [x] metadata 损坏、索引损坏和 Git 回档后的重建流程有明确状态和测试。

## Module 9: RAG 查询与 SearchProvider

**目标**: 为 LLM、SearchNode 和创作总结机制提供少工具强参数的检索能力。若本节与 `创作总结机制(不可删除).md` 冲突，优先实现总结机制。

### 9.1 模块职责

- 统一 RAG 工具 schema，并按总结机制区分 Planner、Detail、Writer、Summarizer 四个独立节点；一个节点就是一个 agent。
- 使用 `operation` 参数承载语义搜索、结构化查询、邻近上下文、摘要查询。
- 创作本地知识查询和 Web/SearchProvider 外部搜索分离，外部搜索结果不自动入库。
- 内置 prompt/display name 资源化，代码不硬编码提示词正文或 UI 显示名。
- 故事段、事件、人物性格变化、人物关系变化、伏笔和章节/阶段总结建立结构化模型。
- 故事段正文只保存 `SourceSpan`，默认查询返回轻量结果，显式要求时才回填正文。
- Writer 可用 `writer-find` 保持创作连续性，可用 `writer-search` 考据现实情况，但不具备 register。
- Writer 正文修改使用 1-based 行号工具，并转换为现有 `DocumentPatch`。
- 混合召回: Qdrant + Tantivy + SQLite，后续在当前创作工具契约下接入真实后端。
- reranker 重排后返回引用。

### 9.2 验收标准

- [x] Planner 暴露 `planner-register`、`planner-find`、`planner-search`。
- [x] Detail 暴露 `detail-find`、`detail-search`。
- [x] Writer 暴露 `writer-find`、`writer-search`、`writer-insert-lines`、`writer-replace-lines`。
- [x] Writer 不暴露 register，search 结果不允许自动注册。
- [x] Planner、Detail、Writer、Summarizer 均建模为独立节点。
- [x] Summarizer 建模为独立节点，但不暴露普通工具集合。
- [x] register 内容使用强类型结构，人物性格、人物关系和伏笔功能可校验。
- [x] find 默认返回 id、摘要、引用、score、来源和 metadata；显式要求才回填正文。
- [x] SearchProvider 结果和本地知识库分离，不自动持久化。
- [x] 故事段、事件、注册项和伏笔支持双向索引。
- [x] Writer 行号 patch 可转换为 UTF-8 byte-range `DocumentPatch`。
- [x] Planner、Detail、Writer、Summarizer 节点上下文可按 agent 特化组装。
- [x] Summarizer 流水线可按故事段、事件、章节、阶段顺序应用结构化草稿。
- [x] 普通模式和 Auto Mode 的确认项初始状态有可测试契约。
- [x] 未落地注册项会生成 Planner 问题队列项。
- [ ] Qdrant + Tantivy + SQLite 真实混合召回接入。
- [ ] SearchNode 工作流节点和前端搜索面板接入。

## Module 10: Skill 系统

**目标**: 支持可复用、可参数化、可沙箱执行的 Skill。

### 10.1 模块职责

- 加载全局 Skill 和项目 Skill，项目优先。
- Skill schema 生成 typed ports。
- 执行 LLM Executor、HTTP Executor、WASM Script Executor。
- HTTP 和 WASM 执行受权限、预算、超时、日志约束。

### 10.2 WASM 安全

- 文件访问必须走授权路径。
- 网络访问默认禁止，除非权限允许。
- 限制运行时间、内存、输出大小。
- 不支持不受控本机脚本。
- Skill 日志必须脱敏，不记录密钥、完整认证头或敏感环境变量。

### 10.3 验收标准

- [ ] 三类 executor 都能在工作流中执行。
- [ ] WASM 越权访问被拒绝。
- [ ] Skill 输出可连接到下游 typed ports。
- [ ] HTTP/WASM 权限拒绝、超时、输出过大和敏感日志脱敏均有测试。

## Module 11: 工作流引擎与审批模块

**目标**: 执行完整 Agent 工作流，支持受限循环、审批、恢复和可观测性。

### 11.1 模块职责

- 执行 typed port 工作流。
- 支持 LLM、Document、Skill、Search、Condition/Eval、Loop/Trigger、Approval、Export 节点。
- 支持受限循环。
- 支持 Pause、Stop、Resume。
- 保存运行状态、节点输出引用、工具调用、审批意见、成本和错误。
- 运行状态持久化必须能恢复 PortValue 引用、artifact 引用、checkpoint、预算消耗和待审批状态。
- 恢复时必须校验引用目标是否存在；缺失时进入可诊断失败状态，而不是重新生成或静默跳过。

### 11.2 Pause / Stop

- Pause: 保存运行态，可继续。
- Stop: 终止当前运行，保留已完成节点、artifact、checkpoint 和 Git 历史。

### 11.3 审批模块

- 节点级配置是否 Auto Mode 自动审批。
- 节点级选择审批提示词。
- 审批 LLM 意见作为输出引脚。

### 11.4 验收标准

- [ ] 受限循环必须有最大轮次、预算、超时和终止条件。
- [ ] 失败后能从最近安全点恢复。
- [ ] Stop 后不能丢失已完成结果。
- [ ] 审批输出可被下游节点使用。
- [ ] Resume 对已完成节点幂等，不重复计费或重复写回。
- [ ] runtime.db 损坏、artifact 缺失、provider 中断和 sidecar 崩溃都有恢复或降级路径测试。

## Module 12: 前端工作区、快捷命令与导出

**目标**: 提供编辑器和节点画布并重的桌面体验。
**本模块特别要求**：本模块需要人工协同开发，写完ui后要根据人工要求不断改进，可以使用网络搜索类似的ui设计，并需要个性化模块

### 12.1 模块职责

- Tauri 2 + React + TypeScript。
- CodeMirror 6 编辑 Markdown、txt、JSON。
- React Flow 节点画布，显示 typed ports。
- 运行面板显示日志、工具调用、成本、Pause/Stop/Resume。
- Git 面板显示存档点、checkpoint、diff、分支图。
- RAG/知识库面板显示索引、冲突、待审队列。
- 支持基础 EPUB/PDF 导出。
- 诊断面板显示性能指标、sidecar 健康、索引进度、慢操作、错误恢复建议和测试环境信息。
- 提供项目初始化、密钥配置、权限配置、恢复流程和导出流程的基础用户引导。

### 12.2 项目管理

- 一次打开一个项目。
- 支持最近项目和项目切换。
- 多项目并行由用户开启多个窗口，不在单窗口内调度。

### 12.3 验收标准

- [ ] 编辑器和节点画布都可作为主工作区。
- [ ] 百万字项目操作不阻塞主 UI。
- [ ] 用户可以查看完整运行日志和成本。
- [ ] 可导出基础 EPUB/PDF。
- [ ] 用户可以查看性能诊断、sidecar 状态、索引重建状态和错误恢复入口。
- [ ] 首次配置、API Key、权限、Git、导出均有基础引导或帮助入口。

## MVP 集成验收场景

1. **百万字项目人工验收**
   - 导入真实百万字项目。
   - 完成索引、检索、写作、patch 写回、Git checkpoint、回档和恢复。

2. **混合 RAG**
   - 同一查询触发向量、全文、结构化召回和 reranker。
   - 返回结果包含来源引用。

3. **受限循环**
   - LLM 生成内容，Eval 判断不通过，Loop 进入下一轮。
   - 达到条件或最大轮次后退出。

4. **Pause / Stop**
   - Pause 后可继续。
   - Stop 后运行终止，但已完成节点、Git、artifact、checkpoint 保留。

5. **审批模块**
   - 普通模式人工审批。
   - Auto Mode 节点级自动审批。
   - 审批 LLM 意见通过输出引脚传给下游。

6. **知识冲突**
   - 新内容与旧设定冲突。
   - 写作 LLM 给出合理性理由。
   - 独立审批 LLM 给出判断。
   - 冲突进入待审队列。

7. **Skill 执行**
   - LLM、HTTP、WASM 三类 Skill 成功执行。
   - WASM 越权访问被拒绝。

8. **成本与 Auto Mode**
   - Auto Mode 在预算内自动执行。
   - 超预算时工作流 Pause。

9. **Git 策略**
   - 文档/工作流/Skill 入库策略可配置。
   - 缓存、索引、成本库默认忽略。

10. **导出**
    - Markdown 项目导出基础 EPUB/PDF。

11. **迁移与恢复**
    - 从旧 schema 项目升级到当前 schema。
    - 迁移失败后可以回滚或重试。
    - 索引损坏、sidecar 崩溃、runtime.db 损坏时有明确恢复或降级路径。

12. **性能与诊断**
    - 百万字数据集完成导入、索引、检索、写回和恢复的性能回归。
    - 前端能查看 sidecar、索引、后台任务、成本和错误状态。
